# KinFrame v0.1 严格审查报告

审查对象：`docs/code_plan_min_v0.1.md` 规定的 v0.1 PRD 转向版实现。

审查结论：当前 v0.1 不能按“严格完成”认定。主体闭环已经基本搭建，包括登录、上传、Worker、fallback slide design、`/showcase`、admin jobs 和备份入口；后端测试在 Docker 中通过，前端 Docker build 也通过。但仍有若干明确违反计划文档的功能缺口和验收风险。

## 已验证结果

1. 后端完整测试在 Docker 后端镜像中通过：`34 passed`。
2. 前端生产构建在 Docker 前端镜像中通过：`pnpm build` 成功。
3. KinFrame Docker 镜像存在：`kinframe-backend-env:stage5`、`kinframe-frontend-env:stage5`。
4. 当前 KinFrame compose 基础设施未运行，因此未将端到端 `just accept-v0-1` 作为通过依据。

## 高优先级问题

### 1. v0.1 showcase 验收脚本不可信

`/api/showcase` 明确依赖登录用户，但 `scripts/v0.1-acceptance.sh` 直接无 cookie 调用：

- `backend/app/api/showcase.py:19`：`GET /api/showcase` 依赖 `get_current_user`。
- `scripts/v0.1-acceptance.sh:42`：直接 `curl "$API_BASE/showcase"`。
- `scripts/v0.1-acceptance.sh:62`：直接 `curl "$FRONTEND_BASE/showcase"`。
- `frontend/pages/showcase.vue:174`：未登录时会跳转 `/login`。

影响：验收脚本可能失败于 401，也可能只验证到登录页/跳转结果，无法证明“上传照片最终能在 `/showcase` 播放”。这违反阶段 0、阶段 4、阶段 6 的验收口径。

### 2. 上传分类仍然强制必填

计划要求“分类可留空，留空时等待 AI 或 fallback 分类”，并要求“未选择分类的照片不会导致上传失败”。

- `docs/code_plan_min_v0.1.md:294`：要求上传表单分类可留空。
- `docs/code_plan_min_v0.1.md:321`：要求未选择分类不导致上传失败。
- `backend/app/api/photos.py:190`：单图上传 `category: Form(...)` 必填。
- `backend/app/api/photos.py:223`：批量上传 `category: Form(...)` 必填。
- `frontend/pages/upload.vue:8`：前端默认分类为 `life`。
- `frontend/pages/upload.vue:114`：前端总是提交 `category`。

影响：阶段 5 未完成，后续 AI 自动分类接入会被当前 API 形态卡住。

## 中优先级问题

### 3. 分类表已新增，但系统仍不是完整可扩展分类

系统新增了 `categories` 表，但多个核心位置仍绑定固定枚举：

- `backend/app/models/photo.py:23`：`photos.category` 仍有限定 `life/travel/photography/pet`。
- `backend/app/schemas/photo.py:10`：Pydantic 类型仍是固定 Literal。
- `backend/app/api/photos.py:57`：上传校验仍走静态集合。

影响：新增分类需要同步改 DB constraint、schema、校验和前端，不符合“可扩展分类”的架构目标。

### 4. Slide Renderer 未按计划过滤未知 layer

计划要求前端校验“过滤未知 layer”，并保证非法 layer 不破坏页面。

- `docs/code_plan_min_v0.1.md:211`：要求过滤未知 layer。
- `frontend/app/slide-renderer/validators/validateSlideDesign.ts:56`：未知 layer type 直接抛错。
- `frontend/app/slide-renderer/components/SlideRenderer.vue:19`：校验失败后整个 design 变为 fallback。

影响：未来 AI 生成的 design 只要包含一个不支持的 layer，就会导致整张 slide 不按预期渲染，而不是保留合法层继续播放。


### 5. 前端渲染验收缺失

计划要求 Nuxt build 通过、至少 3 份示例 design JSON 可渲染，并能通过 Playwright 或验收脚本看到照片。

- `docs/code_plan_min_v0.1.md:223`：定义 Slide Renderer 验收指标。
- `frontend/package.json:6`：`test` 仍是占位命令。
- 当前没有前端 renderer/showcase 自动化测试。

影响：前端构建可通过，但无法证明 renderer 在不同 design、移动端和异常 layer 下稳定工作。

## 架构符合度评估

整体方向基本符合既定架构：

1. 后端仍保持 FastAPI 分层结构。
2. 上传、对象存储、EXIF、preview、thumbnail、Worker 流程可复用。
3. `slide_designs` 表和 fallback generator 已接入 Worker。
4. `/showcase` 已成为新的主体验页面。
5. AI 能力尚未伪装成已完成，当前边界基本清楚。

主要偏差集中在：

1. 分类系统仍是枚举驱动，不是真正数据驱动。
2. 验收脚本没有按登录态真实验证 showcase。
3. Renderer 的容错策略偏脆弱，不利于后续 AI 设计生成。
4. 上传接口没有为“分类可空、AI 后置分类”预留真实行为。

## 遗留大坑

1. 验收脚本可能给出错误结论，后续迭代容易在“以为已通过”的基础上继续叠功能。
2. 分类扩展如果不尽早改成数据驱动，后续每加一个分类都会牵动 DB、后端 schema、校验和前端。
3. AI slide design 接入后，未知 layer 或轻微 schema 偏差会导致整张 slide fallback，影响演示稳定性。
4. 前端缺少 renderer 测试，视觉和交互问题只能靠人工发现。
5. `showcase` 没有真实端到端验收，无法保证“上传 -> Worker -> slide design -> 播放”的完整链路持续可用。

## 建议修复顺序

1. 先修复 `scripts/v0.1-acceptance.sh`：登录后用 cookie 验证 `/api/showcase` 和 `/showcase`，并确认返回内容中包含真实照片和 slide design。
2. 修改上传接口和前端表单，允许分类为空，并在无 AI 时使用明确 fallback 分类策略。
3. 将分类校验从固定枚举逐步迁移到 `categories` 表。
4. 调整前端 renderer 校验逻辑：未知 layer 应过滤，合法 layer 应继续渲染。
5. 为 Slide Renderer 增加最小前端测试或验收样例，覆盖 3 个模板、非法 layer、移动端尺寸。
6. 补充 `/showcase` 每分类数量返回和展示。

## 总体判断

当前 v0.1 是一个可继续推进的 PRD 转向版基础，但还不是严格意义上的完成版。它已经完成了大部分底座和主体验雏形，但阶段 5、阶段 3、阶段 4、阶段 6 均存在未完成项或验收缺口。进入下一阶段前，建议先修复上述 P0/P1 问题，尤其是验收脚本、分类可空和 renderer 容错，否则后续 AI 能力接入会放大这些问题。
