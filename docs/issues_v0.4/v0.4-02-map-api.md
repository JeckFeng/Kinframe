# v0.4-02: Map Data API

## Type

AFK — can be implemented and merged without human interaction.

## Blocked by

None — can start immediately. The frontend map page (v0.4-03) can be developed in parallel using mock data.

## What to build

Add a backend API endpoint that returns geocoded photo data suitable for rendering markers on a map. Each record includes the photo ID, GPS coordinates, location text, category, preview URL, and caption — everything a map marker needs.

This is a backend-only slice: FastAPI route → service query → JSON response → tests.

### Key design decisions

1. **Endpoint**: `GET /api/map/photos` — returns all photos that have GPS coordinates and a successful geocoding status.
2. **Filtering**: Supports optional `?category=` query parameter to filter by category slug.
3. **Pagination**: Not needed for v0.4. A family photo collection is expected to be small (hundreds, not millions). Return all matching records.
4. **Authentication**: Requires valid session (same auth as all other API endpoints). No admin-only restriction — all logged-in users can see the map.
5. **Response format**: A flat list of map-ready photo records, sorted by `taken_at` descending.

### Response schema

```json
{
  "photos": [
    {
      "photo_id": "uuid-string",
      "preview_url": "https://minio-presigned-url...",
      "thumbnail_url": "https://minio-presigned-url...",
      "category": "life",
      "gps_lat": 30.5728,
      "gps_lng": 104.0668,
      "location_name": "四川省成都市武侯区天府大道",
      "location_city": "成都市",
      "location_region": "四川省",
      "location_country": "中国",
      "location_district": "武侯区",
      "final_caption": "一家人在一起就是最好的时光",
      "taken_at": "2025-03-15T14:30:00+08:00"
    }
  ]
}
```

Only photos meeting ALL of these conditions are included:
- `status == 'ready'`
- `gps_lat IS NOT NULL` and `gps_lng IS NOT NULL`
- `geocoding_status == 'success'`

### Implementation plan

#### 1. Pydantic schema (`backend/app/schemas/photo.py`)

Add a new response schema:

```python
class MapPhotoItem(BaseModel):
    photo_id: str
    preview_url: str
    thumbnail_url: str
    category: str
    gps_lat: float
    gps_lng: float
    location_name: str | None
    location_city: str | None
    location_region: str | None
    location_country: str | None
    location_district: str | None
    final_caption: str | None
    taken_at: datetime | None

class MapPhotosResponse(BaseModel):
    photos: list[MapPhotoItem]
```

#### 2. API route (`backend/app/api/photos.py` or new `backend/app/api/map.py`)

Add a new route. Prefer adding to the existing `photos.py` router for simplicity:

```python
@router.get("/map/photos", response_model=MapPhotosResponse)
def get_map_photos(
    category: str | None = Query(None),
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user),
    storage: ObjectStorage = Depends(get_object_storage),
):
    """Return geocoded photos for map display."""
    ...
```

Or create a separate `backend/app/api/map.py` router and register it in `main.py`. Either approach is fine — use a separate file if the map logic grows.

#### 3. Service query (`backend/app/services/photos.py` or inline)

Query the `photos` table:

```sql
SELECT * FROM photos
WHERE status = 'ready'
  AND gps_lat IS NOT NULL
  AND gps_lng IS NOT NULL
  AND geocoding_status = 'success'
  AND (category = :category OR :category IS NULL)
ORDER BY taken_at DESC
```

Generate presigned URLs for `preview_url` and `thumbnail_url` using the existing `storage.presigned_get_url()` method (900s default expiry from `app/services/storage.py`).

#### 4. Tests (`backend/tests/test_map.py`)

Test cases:
- Returns empty list when no photos have GPS data
- Returns only photos with `geocoding_status == 'success'`
- Filters by category correctly
- Excludes photos without GPS coordinates
- Excludes photos with `status != 'ready'`
- Unauthenticated request returns 401
- Response includes valid presigned URLs

### Geocoding prerequisite note

The map endpoint requires geocoded data. If the development database has no geocoded photos:
1. Enable `GEOCODING_ENABLED=true` in `.env`
2. Upload photos with GPS EXIF data, or use the existing admin API to set GPS coordinates on test photos
3. Run the worker to process geocoding jobs

The acceptance script `scripts/v0.3-acceptance.sh` already creates test users and uploads photos — the photos won't have GPS coordinates by default. For map testing, either:
- Manually update a few photo records with GPS coords + geocoding data via the admin API
- Add a small seed script that inserts test geocoding data
- Write the test to seed its own data
