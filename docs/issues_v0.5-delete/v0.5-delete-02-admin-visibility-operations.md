# v0.5-delete-02: Admin Visibility Operations

**Type**: AFK  
**Blocked by**: v0.5-delete-01

## What to build

Expose the new showcase-visibility state in the admin workflow so admins can quickly find hidden photos, identify why a photo is absent from `showcase`, and toggle visibility for any photo without leaving the operations surface.

This slice should deliver one complete admin path:

1. Admin opens `/admin/photos`
2. Admin can see which photos are hidden from `showcase`
3. Admin filters by visibility state
4. Admin toggles hide/unhide for any photo
5. Photo detail and showcase reflect the result

## Behavior

- Add a visibility column or badge in `/admin/photos`
- Add filtering for:
  - visible in showcase
  - hidden from showcase
- Allow admin hide/unhide actions from:
  - `/admin/photos`
  - `/photo/{id}` admin detail view
- Reuse the same `include_in_showcase` field introduced in issue 01
- Do not mix permanent delete into this slice

## Operational expectations

- Hidden photos should not be treated as errors
- Admin page should make hidden state visually distinct from:
  - geocoding failure
  - AI failure
  - review-needed state
- Admin should not need to inspect raw DB fields to know why a photo is missing from `showcase`

## Acceptance criteria

- [ ] `/admin/photos` shows whether each photo is visible or hidden in showcase
- [ ] `/admin/photos` supports filtering by showcase visibility
- [ ] Admin can hide or unhide any photo from the admin list
- [ ] Admin can hide or unhide any photo from admin photo detail
- [ ] Toggling from admin surfaces updates `showcase` behavior immediately
- [ ] Hidden state is not conflated with AI/job/geocoding failure states
- [ ] Frontend tests cover admin visibility filter and toggle behavior
- [ ] Existing admin tests still pass
