# v0.5-delete-05: Delete Audit + Trash Readiness

**Type**: AFK  
**Blocked by**: v0.5-delete-01, v0.5-delete-03

## What to build

Make delete-related actions auditable and leave a clean architectural seam for a future recycle-bin system. This slice is not a full trash feature. It is the governance layer that makes hide/purge actions explainable and future-proof.

This slice should deliver:

1. Hide/unhide actions create audit records
2. Permanent delete request/success/failure create audit records
3. Admin can inspect what happened without reading raw logs
4. Data model or audit payload preserves enough context to support a later trash flow

## Behavior

- Record audit events for:
  - `photo.hide`
  - `photo.unhide`
  - `photo.delete_requested`
  - `photo.delete_succeeded`
  - `photo.delete_failed`
- Audit payload should include:
  - acting user/admin
  - target photo id
  - previous showcase visibility state where relevant
  - deletion job id where relevant
  - summary/error detail
- Preserve future trash-readiness by keeping enough metadata to answer:
  - who deleted it
  - when it was deleted
  - why a purge failed

## Deliberate non-goals

- No restore-from-trash UI
- No scheduled purge daemon
- No user-facing recycle bin

## Acceptance criteria

- [ ] Hide action writes an audit record
- [ ] Unhide action writes an audit record
- [ ] Permanent delete request writes an audit record
- [ ] Purge success writes an audit record
- [ ] Purge failure writes an audit record with error summary
- [ ] Admin can inspect delete-related audit trail through existing admin flows
- [ ] The resulting data shape leaves a clean path to future recycle-bin work
- [ ] Backend tests cover audit creation for hide and delete flows
