# v0.5-delete-03: Permanent Photo Purge Job Pipeline

**Type**: AFK  
**Blocked by**: None — can start immediately

## What to build

Implement permanent deletion as a backend job pipeline instead of a synchronous API hard-delete. This slice delivers the full backend path for admin-triggered purge:

1. Admin requests permanent deletion
2. System enqueues a `photo_purge` job
3. Worker deletes MinIO objects and DB records
4. Success or failure is visible through job state

This is the core destructive path. It must be safe, diagnosable, and idempotent enough for retry.

## Behavior

- Add a new job type: `photo_purge`
- Add an admin-only endpoint:
  - `POST /api/admin/photos/{photo_id}/delete`
- The endpoint should:
  - verify photo exists
  - verify caller is admin
  - enqueue purge job
  - return `job_id`
- Worker `photo_purge` execution should remove:
  - original object from MinIO
  - preview object from MinIO
  - thumbnail object from MinIO
  - slide design versions
  - photo processing jobs as appropriate
  - photo record itself

## Consistency rules

- Do not implement permanent deletion as one fragile synchronous request
- If object deletion fails, the job must fail with a clear error
- If the job fails, admin must be able to retry
- The system must avoid silent half-delete states

## Scope notes

- This slice is backend-first; UI confirmation can land later
- Full recycle-bin restore is out of scope for this issue

## Acceptance criteria

- [ ] Admin-only delete endpoint exists and enqueues `photo_purge`
- [ ] Non-admin caller cannot enqueue permanent deletion
- [ ] Worker processes `photo_purge` jobs
- [ ] Successful purge removes MinIO original, preview, and thumbnail objects
- [ ] Successful purge removes photo record and associated slide design records
- [ ] Failed purge records a useful job error and does not silently succeed
- [ ] Purge job can be retried after failure
- [ ] Backend tests cover success path, MinIO failure path, and permission rejection
- [ ] Existing worker/job tests still pass
