# v0.3-12: Schema Sharing Between Frontend and Backend

**Type**: AFK
**Blocked by**: None — can start immediately

## What to build

Currently the frontend (TypeScript validator) and backend (Python validator + JSON Schema) maintain separate slide design validation logic. Unify them to use the same `slide_design.schema.json` as the single source of truth.

### Approach

- The canonical schema file lives at a shared location accessible to both frontend and backend
- Backend Python validator already uses `slide_design.schema.json` for JSON Schema validation — ensure it covers all v0.2+ fields
- Frontend TypeScript validator adds a JSON Schema validation pass (using `ajv` or equivalent) before its semantic validation
- Both sides use identical field names, types, ranges, and constraints

### Schema completeness

Verify the shared schema covers all current fields:
- `version`, `photoId`, `templateId`, `canvas`, `templateParams`
- `mediaRefs` (if used)
- `captionPolicy` (captionSource, userMessage, aiCaption)
- `styleTokens` (cssVariables, scopedCss)
- `layers[]` with all layer types and their fields
- `timeline` (takenAt, displayDate, categoryOrderIndex, categoryTotal)
- `renderPolicy` (fallbackTemplateId, allowHtml: false, allowJavaScript: false, allowScopedCss, allowMotion)
- `aiMeta` (provider, model, promptVersion, generatedAt)

### Validation layering

1. JSON Schema validation (structural: required fields, types, ranges) — shared
2. Semantic validation (contextual: templateId exists, layer types in whitelist, CSS variables in whitelist) — language-specific
3. Security sanitization (CSS scrubbing, HTML/JS rejection) — language-specific

Layer 1 must be identical. Layers 2 and 3 use the same config files (templates, layer_primitives, CSS whitelist) but may have language-specific implementations.

## Acceptance criteria

- [ ] Single shared `slide_design.schema.json` covers all v0.2+ slide design fields
- [ ] Backend Python validator uses it for structural validation via `jsonschema` library
- [ ] Frontend TypeScript validator uses it for structural validation via `ajv`
- [ ] Both sides reject the same set of structurally invalid designs
- [ ] Both sides accept the same set of structurally valid designs
- [ ] New test: round-trip — backend validates → frontend validates → consistent results
- [ ] Existing 108 backend + 42 frontend tests still pass
- [ ] CI check: schema file hasn't diverged between frontend/ and backend/ copies
