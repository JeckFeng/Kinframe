# KinFrame 产品需求文档（PRD）

> 文档版本：v1.1  
> 项目名称：KinFrame  
> 项目根目录：`kinframe`  
> 产品定位：AI 驱动的家庭影像艺术化 PPT 放映网站  
> 目标用户：家庭成员、小范围私人用户  
> 核心形态：用户上传照片素材，AI 自动分析照片内容、意境、主题色和元数据，并为每张照片生成一页具有摄影作品集质感的全屏 PPT 式网页幻灯片。

---

## 1. 文档目的

本文档用于明确 KinFrame 项目的产品需求、交互细节、功能边界、AI 工作流、页面结构、用户流程、数据规则和验收标准。

本文档应作为后续开发、设计、测试和验收的基础依据。开发过程中，所有页面、组件、接口和 AI 工作流均应优先满足本文档定义的产品逻辑。

---

## 2. 产品背景

传统家庭相册网站通常采用图片网格、相册列表、文件夹管理、标签检索等形式。这类产品偏向“图片管理”，但缺乏家庭影像的仪式感、艺术感和沉浸式观看体验。

KinFrame 的目标不是复制 Immich、Google Photos 或传统相册管理站，而是将家庭影像做成类似摄影作品网站、艺术展览、PPT 放映和数字家庭影像馆的综合体验。

用户登录后，首先看到的不是图片管理后台，而是一个全屏放映页面：每一页只展示一张照片，每一张照片都拥有独立的背景、布局、文案、时间轴和动效。系统通过 AI 理解照片内容，并生成与照片相匹配的幻灯片设计方案。

## 3. 产品定位

### 3.1 一句话定位

KinFrame 是一个 AI 驱动的家庭影像艺术放映系统，用于将家庭照片自动整理、分类、设计并展示为高审美的全屏 PPT 式影像作品集。

### 3.2 产品关键词

- 家庭影像
- 私有部署
- 全屏放映
- PPT 式展示
- 摄影作品集
- AI 设计
- 图片智能入库
- EXIF 元数据
- 分类播放
- 时间轴叙事
- 隐藏式菜单
- 沉浸式观看

### 3.3 产品不是什么

KinFrame 不是以下类型产品：

1. 不是传统文件夹式图片管理器。
2. 不是以网格浏览为主的图库系统。
3. 不是面向公开互联网的摄影社交网站。
4. 不是面向大量陌生用户注册的平台型产品。
5. 不是复杂的企业级 DAM 数字资产管理系统。
6. 不是简单的图片上传和列表展示系统。

## 4. 目标用户

### 4.1 家庭普通用户

典型用户包括：用户本人、伴侣、父母、兄弟姐妹等小范围家庭成员。

使用特点：

- 希望上传和观看家庭照片。
- 不关心复杂管理功能。
- 更重视照片播放体验是否美观、自然、易用。
- 可能使用 iPhone、Android 手机或电脑访问。
- 希望随时随地上传照片。

### 4.2 管理员用户

管理员通常是项目所有者。

职责包括：

- 创建和管理家庭成员账号。
- 管理照片分类。
- 查看上传记录。
- 删除、隐藏或修正照片信息。
- 处理 AI 分类错误或文案不合适的问题。
- 维护系统部署、备份和迁移。

## 5. 产品目标

### 5.1 第一阶段产品目标

实现一个可登录、可上传、可分类、可解析 EXIF、可生成 AI 幻灯片设计、可全屏 PPT 放映的私有家庭影像网站。

### 5.2 核心目标

1. 用户登录后进入全屏 PPT 式影像放映主页。
2. 每一页幻灯片只展示一张照片。
3. 顶部菜单栏和左侧分类栏默认隐藏，仅在用户触发时显示。
4. 用户可通过键盘方向键切换照片和分类。
5. 照片按分类组织为不同 PPT。
6. 同一分类内照片按拍摄时间或上传时间形成时间轴。
7. 用户上传照片时可填写留言，也可允许 AI 自动生成文案。
8. AI 能识别图片内容、主题、主题色和意境，并生成幻灯片设计 JSON。
9. 前端根据 AI 设计 JSON 渲染每一页幻灯片。
10. 系统支持外网访问和私有化部署。

## 7. 核心产品形态

### 7.1 主体验形态

用户登录后，默认进入全屏 PPT 放映页面。

页面视觉特征：

- 全屏展示。
- 无明显后台管理感。
- 一页一张照片。
- 每页具有独立艺术设计。
- 背景、布局、文案、时间轴均与当前照片匹配。
- 菜单默认隐藏。
- 用户通过方向键在照片和分类之间切换。

### 7.2 内容组织方式

KinFrame 采用“分类 PPT + 时间轴”的组织方式。

默认分类：

1. 生活照。
2. 摄影照。
3. 宠物照。

每个分类对应一套独立 PPT：

- 生活照 PPT。
- 摄影照 PPT。
- 宠物照 PPT。

每个 PPT 内包含多张照片，每张照片是一页幻灯片。

## 8. 信息架构

### 8.1 页面结构

建议页面包括：

```text
/login                    登录页
/showcase                 全屏 PPT 放映主页
/upload                   照片上传页
/gallery                  图片总览页
/gallery/:category        分类图片总览页
/photo/:id                图片详情页
/admin                    管理后台首页
/admin/photos             图片管理
/admin/categories         分类管理
/admin/users              用户管理
/admin/jobs               AI 处理任务管理
/settings/profile         用户个人信息
```

### 8.2 默认登录后跳转

用户登录成功后，应默认跳转到：

```text
/showcase
```

而不是跳转到图片总览或后台管理页面。

## 9. 角色与权限

### 9.1 角色类型

#### 普通用户

权限：

- 登录系统。
- 浏览所有可见照片。
- 上传照片。
- 为自己上传的照片填写留言。
- 选择是否启用 AI 文案。
- 查看图片总览。
- 查看照片详情。

#### 管理员

权限：

- 拥有普通用户全部权限。
- 创建用户。
- 禁用用户。
- 删除或隐藏照片。
- 修改照片分类。
- 修改照片文案。
- 重新触发 AI 设计生成。
- 管理分类。
- 查看处理任务。

### 9.2 鉴权原则

1. 未登录用户不能访问任何照片页面。
2. 未登录用户只能访问登录页。
3. 不开放自由注册。
4. 用户账号由管理员创建。
5. 所有图片访问必须经过权限校验。
6. MinIO bucket 不公开。
7. 前端加载图片时应通过后端获取短期有效访问 URL。

## 11. 全屏 PPT 放映主页需求

### 11.1 页面定位

`/showcase` 是 KinFrame 的核心页面。

该页面不是普通图片详情页，而是沉浸式家庭影像播放空间。

### 11.2 页面基本布局

页面由以下层级组成：

```text
全屏容器 ShowcasePage
├── 背景层 BackgroundLayer
├── 图片层 PhotoLayer
├── 文案层 CaptionLayer
├── 元信息层 MetaLayer
├── 底部时间轴 TimelineBar
├── 顶部隐藏菜单 TopHiddenNav
├── 左侧隐藏分类栏 LeftCategoryNav
├── 键盘导航控制 KeyboardNavigator
└── 加载与错误状态层 StatusLayer
```

### 11.3 一页一图规则

1. 每一页 PPT 只能展示一张主照片。
2. 主照片是该页唯一核心视觉主体。
3. 不允许在一页中同时展示多张主照片。
4. 背景可以使用当前照片的模糊扩展、渐变、纯色或 AI 生成的主题色背景，但不能喧宾夺主。
5. 文案、时间轴、元信息均服务于当前照片。

### 11.4 全屏展示规则

1. 页面默认占满浏览器视口。
2. 不出现传统网页滚动条。
3. 不出现固定的顶部导航栏。
4. 不出现固定侧边栏。
5. 页面应支持桌面端 16:9、16:10、21:9 显示器。
6. 页面应兼容手机竖屏，但第一版可优先保证桌面端体验。

## 12. 顶部隐藏菜单栏需求

### 12.1 位置

顶部隐藏菜单栏位于浏览器窗口正上方。

默认隐藏在页面顶部，不占用视觉空间。

### 12.2 呼出方式

顶部菜单栏的呼出方式包括：

1. 鼠标移动到窗口顶部触发区域时自动浮现。
2. 触发区域建议为窗口顶部 0 至 64px 区域。
3. 用户按下 `M` 键时可以切换菜单显示状态，作为辅助快捷方式。
4. 移动端可通过点击屏幕顶部区域或右上角轻触区域呼出。

### 12.3 隐藏方式

顶部菜单栏隐藏方式包括：

1. 鼠标离开顶部菜单栏区域后自动隐藏。
2. 鼠标离开后建议延迟 600ms 至 1200ms 再隐藏，避免误触。
3. 用户按下 `Esc` 键时立即隐藏。
4. 用户点击页面空白区域时隐藏。
5. 当菜单中有下拉框、确认框或上传弹窗打开时，菜单不得自动隐藏。

### 12.4 视觉样式

顶部菜单栏应符合以下视觉原则：

1. 半透明。
2. 毛玻璃效果。
3. 不使用重色块。
4. 不破坏照片沉浸感。
5. 菜单浮现时使用轻微下滑和淡入动画。
6. 菜单隐藏时使用淡出和上移动画。
7. 高度建议为 56px 至 72px。
8. 菜单背景可根据当前幻灯片主题色自动调整透明度和明暗。

### 12.5 菜单项布局

顶部菜单栏建议分为左、中、右三区。

#### 左侧区域

可放置：

- KinFrame 产品标识。
- 当前分类名称。
- 当前照片拍摄年份。

#### 中间区域

可放置主要功能入口：

- 上传照片。
- 图片总览。
- 当前照片详情。
- 重新生成当前页设计，管理员可见。

#### 右侧区域

可放置用户相关功能：

- 当前用户头像或用户名。
- 个人设置。
- 管理后台，管理员可见。
- 退出登录。

### 12.6 第一版顶部菜单项

第一版必须包含：

1. 上传照片。
2. 图片总览。
3. 用户信息。
4. 退出登录。

管理员额外显示：

1. 管理后台。
2. 重新生成当前页 AI 设计。

### 12.7 后续适合放在顶部菜单栏的功能

后续可以放入顶部菜单栏的功能包括：

1. 搜索照片。
2. 切换播放模式。
3. 自动播放开关。
4. 音乐开关。
5. 当前照片详情。
6. 下载原图，按权限控制。
7. 收藏当前照片。
8. 分享家庭内部链接。
9. 重新生成文案。
10. 重新生成页面风格。

### 12.8 不适合放在顶部菜单栏的功能

以下功能不应直接放在顶部菜单栏作为一级入口：

1. 数据库管理。
2. 系统日志。
3. 复杂备份恢复操作。
4. 低频开发者配置。
5. 大量分类管理细项。
6. AI 提示词编辑器。

这些功能应放入管理后台。

### 12.9 验收标准

1. 默认进入 `/showcase` 时顶部菜单不可见。
2. 鼠标移动到窗口顶部 64px 范围内时菜单出现。
3. 鼠标离开后菜单自动隐藏。
4. 菜单出现和隐藏有平滑动画。
5. 菜单不会遮挡主要照片主体过多。
6. 上传、图片总览、用户信息、退出登录入口可用。
7. 管理员可以看到管理后台入口。
8. 普通用户不能看到管理员专属入口。

## 13. 左侧隐藏分类栏需求

### 13.1 位置

左侧隐藏分类栏位于浏览器窗口最左侧。

默认隐藏，仅显示极弱提示或完全隐藏。

### 13.2 呼出方式

左侧分类栏的呼出方式包括：

1. 鼠标移动到窗口左侧触发区域时自动浮现。
2. 触发区域建议为窗口左侧 0 至 64px 区域。
3. 用户按下 `C` 键时可以切换分类栏显示状态。
4. 用户通过键盘 `↑` / `↓` 切换分类时，分类栏应短暂显示当前分类状态。
5. 移动端可通过从左侧边缘轻扫呼出。

### 13.3 隐藏方式

1. 鼠标离开左侧分类栏区域后自动隐藏。
2. 鼠标离开后建议延迟 600ms 至 1200ms 再隐藏。
3. 按下 `Esc` 键立即隐藏。
4. 点击页面空白区域隐藏。
5. 分类切换动画完成后，分类栏可在 2 秒后自动隐藏。

### 13.4 分类栏内容

第一版分类栏包括：

1. 生活照。
2. 摄影照。
3. 宠物照。

每个分类项展示：

- 分类名称。
- 当前分类照片数量。
- 当前是否正在播放。
- 可选的分类图标。

### 13.5 当前分类状态

当前正在播放的分类需要明显标识。

标识方式可以包括：

1. 高亮文字。
2. 左侧细线。
3. 小圆点。
4. 当前分类名称放大。
5. 与当前幻灯片主题色协调的强调色。

### 13.6 分类可扩展规则

1. 分类不应在前端硬编码。
2. 分类应从后端接口读取。
3. 后续管理员可新增分类。
4. 每个分类应包含 `code`、`name`、`sort_order`、`is_active`。
5. 停用分类不在左侧播放栏显示。

### 13.7 验收标准

1. 默认状态下左侧分类栏不可见或极弱提示。
2. 鼠标移动到左侧触发区域时分类栏出现。
3. 鼠标离开后分类栏自动隐藏。
4. 分类栏显示三类默认分类。
5. 当前分类有清晰视觉标识。
6. 点击分类项可以切换到该分类 PPT。
7. 使用 `↑` / `↓` 切换分类时，左侧分类栏能同步更新。

## 14. PPT 放映导航需求

### 14.1 键盘方向键规则

KinFrame 的核心放映规则如下：

```text
← / → ：在当前分类 PPT 内切换上一张 / 下一张照片
↑ / ↓ ：切换到其他分类 PPT
```

具体规则：

- `→`：播放当前分类中的下一张照片。
- `←`：播放当前分类中的上一张照片。
- `↓`：切换到下一个分类 PPT。
- `↑`：切换到上一个分类 PPT。

### 14.2 同类照片切换规则

1. 同一分类内照片按时间轴顺序排列。
2. 时间轴顺序默认从旧到新。
3. `→` 进入时间更晚的下一张。
4. `←` 返回时间更早的上一张。
5. 到达最后一张后继续按 `→` 时，可选择循环到第一张。
6. 到达第一张后继续按 `←` 时，可选择循环到最后一张。
7. 第一版建议开启循环播放。

### 14.3 分类切换规则

默认分类顺序：

```text
生活照 → 摄影照 → 宠物照
```

`↓`：按上述顺序向下切换。

`↑`：按上述顺序向上切换。

到达最后一个分类后继续按 `↓`：

- 第一版建议循环回第一个分类。

到达第一个分类后继续按 `↑`：

- 第一版建议循环到最后一个分类。

### 14.4 分类切换后的照片定位规则

分类切换后，当前照片应按以下优先级定位：

1. 如果该分类上次有播放位置，则恢复该分类上次播放的照片。
2. 如果没有上次播放位置，则进入该分类中时间轴最新的一张照片。
3. 如果该分类没有照片，则显示空分类提示页。

## 15. 底部时间轴需求

### 15.1 位置

时间轴位于每一页 PPT 的底部。

位置应接近窗口底部，但不能贴边。

建议距离底部 32px 至 64px。

### 15.2 视觉要求

时间轴应具有唯美、轻量、克制的视觉效果。

不应像传统播放器进度条一样粗重。

可采用：

1. 细线。
2. 小圆点。
3. 年份刻度。
4. 柔和光点。
5. 半透明文字。
6. 低饱和度颜色。
7. 与当前幻灯片主题色相协调的强调色。

### 15.3 内容组成

时间轴至少展示：

1. 当前照片拍摄时间。
2. 当前照片在该分类时间序列中的位置。
3. 分类内时间范围。

可选展示：

1. 年份。
2. 月份。
3. 地点。
4. 当前照片序号。
5. 当前分类照片总数。

### 15.4 时间来源规则

时间字段优先级：

1. EXIF `DateTimeOriginal`。
2. EXIF `CreateDate`。
3. 用户手动修正的拍摄时间。
4. 上传时间。

如果照片没有 EXIF 拍摄时间：

- 系统使用上传时间。
- 在照片详情中标记“使用上传时间”。
- 时间轴仍应正常工作。

### 15.5 时间轴排序规则

同一分类中：

1. 默认按拍摄时间升序排列。
2. 没有拍摄时间的照片按上传时间排序。
3. 时间完全相同的照片按上传时间排序。
4. 用户后续可在后台修改拍摄时间。

### 15.6 时间轴交互

第一版基础交互：

1. 当前照片所在位置用高亮点表示。
2. 用户可看到当前照片处于分类时间轴的哪个位置。

后续增强交互：

1. 点击时间轴节点跳转到对应照片。
2. 拖动时间轴快速浏览。
3. 按年份折叠。
4. 按季节展示。

### 15.7 验收标准

1. 每页 PPT 下方都显示时间轴。
2. 时间轴不会遮挡照片主体。
3. 时间轴能显示当前照片时间。
4. 切换照片时时间轴同步更新。
5. 切换分类时时间轴范围同步更新。
6. 没有 EXIF 时间的照片仍能显示在时间轴上。
7. 时间轴样式与当前幻灯片主题色协调。

1. 图片分类。

5. 是否立即加入放映。

用户可以手动选择分类：

- 生活照。
- 摄影照。
- 宠物照。

如果用户没有选择分类：

- 系统调用 AI 自动分类。

如果用户选择了分类：

- 用户选择优先。
- AI 仍可生成分类建议，但不得覆盖用户选择，除非用户明确选择“允许 AI 自动修正分类”。

### 16.7 文案生成规则

文案字段包括：

1. 用户留言 `user_message`。
2. AI 文案 `ai_caption`。
3. 最终展示文案 `final_caption`。

规则：

```text
如果用户填写了留言：
    final_caption = user_message
    AI 不生成主文案，或生成但不展示。

如果用户没有填写留言，并勾选“AI 生成文案”：
    AI 生成 ai_caption
    final_caption = ai_caption

    final_caption = null
    幻灯片不展示主文案。
```

上传后照片进入处理流程：

```text
uploaded → processing → analyzed → design_generated → ready
```

如果失败：

```text
failed
```

失败时应允许管理员重新处理。

### 16.9 上传成功反馈

用户上传成功后，系统应提示：

1. 照片已上传。
2. 正在解析照片信息。
3. 正在生成幻灯片设计。
4. 完成后将自动加入对应分类 PPT。

### 16.10 验收标准

2. 用户可以上传支持格式的图片。
3. 上传后原图保存到 MinIO。

5. 上传后进入后台处理任务。

7. AI 文案开关规则正确。
8. 上传完成后照片最终可在 PPT 中播放。

## 19. AI 幻灯片设计与渲染能力层需求

### 19.1 基本原则

KinFrame 的 AI 幻灯片设计不采用“AI 直接生成完整网页代码并运行”的方式。

系统应采用以下混合方案：

```text
基础模板体系
+ 模板参数化
+ 图层混合
+ AI 生成式 CSS
```

该方案的核心目标是：在保证安全、稳定、可维护的前提下，尽可能释放 AI 的审美设计能力，让每一张照片都能形成具有差异化、高级感和摄影作品集气质的全屏 PPT 页面。

### 19.2 四层设计能力架构

KinFrame 的单页幻灯片设计由四个能力层共同组成。

| 能力层 | 作用 | 主要控制内容 | 是否由 AI 生成 |
|---|---|---|---|
| 基础模板体系 | 决定页面大体骨架 | 页面结构、主图区域、文案区域、时间轴大致位置 | AI 选择模板，但不生成模板代码 |
| 模板参数化 | 调整模板内部布局 | 主图位置、主图尺寸、文案区域、元信息区域、时间轴位置 | AI 生成参数 |
| 图层混合 | 添加艺术视觉元素 | 蒙版、形状、光斑、渐变层、纹理层、装饰线条、文案底板 | AI 生成图层配置 |
| AI 生成式 CSS | 控制视觉质感 | 主题色、透明度、阴影、滤镜、字体气质、过渡动效、背景氛围 | AI 生成受限 CSS Tokens |

四层之间必须有清晰边界：

```text
JSON 控制结构、布局、位置、大小、图层和语义。
CSS Tokens 控制颜色、质感、滤镜、阴影、透明度和动效细节。
```

不得出现 JSON 和 CSS 同时控制同一个核心布局字段的情况。

### 19.3 设计文件体系

为支撑上述能力，系统至少需要定义以下 JSON 文件或 JSON 数据结构。

#### 19.3.1 `slide_templates.json`

用途：定义 KinFrame 内置的基础模板体系。

该文件描述每个模板的名称、适用分类、适用照片类型、默认布局区域、允许调整的参数范围和模板能力边界。

示例结构：

```json
{
  "version": "1.0",
  "templates": [
    {
      "id": "cinematic_fullscreen",
      "name": "电影感全屏",
      "description": "适合风景、旅行摄影、氛围感较强的照片。",
      "preferredCategories": ["travel", "life"],
      "aspectRatio": "16:9",
      "defaultRegions": {
        "imageArea": { "x": 0.06, "y": 0.08, "width": 0.88, "height": 0.78 },
        "captionArea": { "x": 0.12, "y": 0.72, "width": 0.42, "height": 0.14 },
        "metaArea": { "x": 0.12, "y": 0.86, "width": 0.36, "height": 0.05 },
        "timelineArea": { "x": 0.12, "y": 0.92, "width": 0.76, "height": 0.04 }
      },
      "allowedParams": {
        "imageFit": ["cover", "contain"],
        "captionPosition": ["bottom_left", "bottom_right", "center_bottom"],
        "timelineVariant": ["minimal_line", "soft_glow", "year_marks"],
        "imageRadius": ["none", "small", "medium", "large"]
      },
      "maxExtraLayers": 8
    }
  ]
}
```

第一版建议内置模板：

1. `cinematic_fullscreen`：电影感全屏。
2. `gallery_center`：艺术馆居中展示。
3. `magazine_left`：杂志式左图右文。
4. `magazine_right`：杂志式右图左文。
5. `poetic_landscape`：诗意风景。
6. `warm_memory`：温暖家庭记忆。
7. `pet_portrait`：宠物肖像。
8. `minimal_white`：极简白底。
9. `dark_exhibition`：暗色展厅。
10. `soft_blur_background`：柔焦背景。
11. `film_negative`：胶片感。
12. `quiet_album`：安静影集。

#### 19.3.2 `layer_primitives.json`

用途：定义前端 Slide Renderer 支持的图层类型、字段、默认值和取值范围。

#### 19.3.4 `slide_design.schema.json`

用途：定义单张幻灯片设计 JSON 的校验规则。

所有 AI 生成的 `slide_designs.json` 数据必须先通过 schema 校验，再进入数据库和前端渲染器。

校验内容至少包括：

1. 模板 ID 是否存在。
2. 坐标是否在 0 到 1 范围内。
3. 宽高是否大于 0。
4. 透明度是否在 0 到 1 范围内。
5. 颜色是否为合法 HEX、RGB、RGBA、HSL 或 CSS 变量引用。
6. 图层数量是否超过模板允许上限。
7. zIndex 是否在允许范围内。
8. CSS Tokens 是否在白名单内。
9. 不允许出现任意 HTML、JavaScript 或未授权 CSS 选择器。

### 19.4 JSON 与 CSS 的边界

为避免冲突，必须遵守以下边界。

#### 19.4.1 JSON 负责控制

1. 使用哪个模板。
2. 元素是否存在。
3. 元素类型。
4. 元素位置。
5. 元素大小。
6. 元素图层顺序。
7. 元素语义角色。
8. 图片 fit 模式。
9. 文案区域。
10. 时间轴区域。
11. 蒙版区域。
12. 形状区域。
13. 动画类型。
14. 文案来源。

#### 19.4.2 CSS Tokens 负责控制

1. 颜色。
2. 透明度。
3. 阴影。
4. 模糊。
5. 滤镜。
6. 字体气质。
7. 字重。
8. 字距。
9. 行高。
10. 圆角数值。
11. 背景质感。
12. 纹理强度。
13. 动效时长。
14. 动效缓动曲线。

#### 19.4.3 CSS Tokens 禁止控制

1. 主图坐标。
2. 主图宽高。
3. 文案坐标。
4. 文案容器宽高。
5. 时间轴坐标。
6. 图层 zIndex。
7. 顶部隐藏菜单位置。
8. 左侧分类栏位置。
9. 路由结构。
10. 登录状态。
11. 任意 JavaScript 行为。
12. 任意全局选择器。

### 19.5 坐标、大小与响应式规则

所有幻灯片元素的位置和大小默认使用 0 到 1 的归一化坐标系。

```json
"position": {
  "x": 0.12,
  "y": 0.18,
  "width": 0.58,
  "height": 0.64
}
```

含义：

- `x`：元素左上角距离画布左侧的比例。
- `y`：元素左上角距离画布顶部的比例。
- `width`：元素占画布宽度的比例。
- `height`：元素占画布高度的比例。

规则：

1. `x`、`y`、`width`、`height` 默认范围为 `0` 到 `1`。
2. `width` 和 `height` 必须大于 `0`。
3. 元素可以轻微超出画布，用于光斑或背景纹理，但必须显式声明 `allowOverflow: true`。
4. 主图、文案、时间轴不得超出安全区域。
5. 移动端可以由前端渲染器基于模板定义进行自适应重排。

### 19.6 填充、渐变与颜色规则

图层的填充统一使用 `fill` 字段。

支持类型：

1. `solid`：纯色。
2. `linearGradient`：线性渐变。
3. `radialGradient`：径向渐变。
4. `imageBlur`：基于当前照片的模糊背景。
5. `noise`：颗粒纹理。
6. `transparent`：透明。

线性渐变示例：

```json
{
  "fill": {
    "type": "linearGradient",
    "angle": 135,
    "stops": [
      { "color": "#0F172A", "opacity": 1, "position": 0 },
      { "color": "#D6A86C", "opacity": 0.75, "position": 1 }
    ]
  }
}
```

规则：

1. 每个渐变至少包含 2 个 stops。
2. `position` 范围为 `0` 到 `1`。
3. `opacity` 范围为 `0` 到 `1`。
4. 渐变颜色不得超过 5 个 stops，避免视觉混乱。
5. AI 可以参考图片主色，但最终颜色应低饱和、协调、克制。

### 19.7 蒙版规则

蒙版用于增强文案可读性、制造层次、控制视觉焦点。

支持蒙版类型：

1. `linearFade`：线性渐隐。
2. `radialFade`：径向渐隐。
3. `rect`：矩形遮罩。
4. `roundedRect`：圆角矩形遮罩。
5. `ellipse`：椭圆遮罩。

蒙版字段示例：

```json
{
  "type": "mask",
  "id": "caption_readability_mask",
  "zIndex": 25,
  "shape": "linearFade",
  "position": { "x": 0.48, "y": 0.56, "width": 0.46, "height": 0.28 },
  "style": {
    "fill": {
      "type": "linearGradient",
      "angle": 90,
      "stops": [
        { "color": "#000000", "opacity": 0.42, "position": 0 },
        { "color": "#000000", "opacity": 0, "position": 1 }
      ]
    },
    "blur": 20,
    "blendMode": "multiply"
  }
}
```

规则：

1. 蒙版不得遮挡照片主体超过合理范围。
2. 蒙版主要用于背景层和文案增强，不应成为主视觉。
3. 蒙版透明度建议不超过 `0.65`。
4. 每页蒙版数量第一版不超过 4 个。

### 19.8 阴影规则

阴影既可由 JSON 定义，也可通过 CSS Token 引用。

推荐方式：

```json
"shadow": "var(--kf-photo-shadow)"
```

需要精确控制时可使用结构化对象：

```json
"shadow": {
  "type": "soft",
  "x": 0,
  "y": 32,
  "blur": 80,
  "spread": 0,
  "color": "rgba(0,0,0,0.38)"
}
```

边界规则：

1. 阴影位置和模糊强度属于视觉质感，优先由 CSS Tokens 控制。
2. 阴影是否存在属于图层结构，归 JSON 控制。
3. 阴影不得造成图片边缘脏污或过重压暗。

### 19.9 字体与文案排版规则

文案图层使用 `text` 类型。

字段包括：

1. `content`：文本内容。
2. `position`：文本容器位置和大小。
3. `fontFamily`：字体族。
4. `fontSize`：字体大小。
5. `fontWeight`：字体粗细。
6. `direction`：文字方向。
7. `textAlign`：水平对齐。
8. `verticalAlign`：垂直对齐。
9. `letterSpacing`：字距。
10. `lineHeight`：行高。
11. `writingMode`：书写模式。
12. `color`：文字颜色。
13. `opacity`：透明度。

文字方向支持：

1. `horizontal`：横排。
2. `vertical-rl`：竖排，从右到左。
3. `vertical-lr`：竖排，从左到右。

规则：

1. 用户留言优先于 AI 文案。
2. 文案不应遮挡人脸、宠物脸、照片主体或关键景物。
3. 文案建议不超过两行。
4. AI 文案建议长度 10 至 35 个汉字。
5. 字体大小可使用 `clamp()`，但必须在 CSS 白名单允许范围内。
6. 文案可竖排，但默认横排。

### 19.10 图层混合规则

每一页幻灯片由多个图层组成。

图层必须包含：

1. `id`
2. `type`
3. `zIndex`
4. `position`，部分全屏背景层可省略并默认全屏。
5. `style`

图层顺序规则：

```text
0-9：背景和纹理
10-19：主照片
20-29：蒙版和背景增强
30-39：形状和装饰
40-49：文案和元信息
50-59：时间轴
60+：临时交互层，仅前端系统使用，AI 不可生成
```

AI 不得生成 `zIndex >= 60` 的图层。

### 19.11 AI 生成式 CSS 规则

AI 生成式 CSS 第一版仅允许输出 CSS Variables。

输出位置：

```json
"styleTokens": {
  "cssVariables": {
    "--kf-caption-color": "rgba(248,244,234,0.92)",
    "--kf-photo-shadow": "0 32px 80px rgba(0,0,0,0.38)"
  }
}
```

P1 阶段可允许受限 scoped CSS：

```json
"styleTokens": {
  "scopedCss": ".kf-slide-caption { text-wrap: balance; }"
}
```

但 scoped CSS 必须满足：

1. 只能使用白名单选择器。
2. 只能使用白名单属性。
3. 不能控制核心布局属性。
4. 不能出现 `@import`。
5. 不能出现 `url()` 外部资源。
6. 不能出现 `position`、`left`、`top`、`width`、`height`、`z-index` 等布局字段。
7. 不能影响顶部菜单栏、左侧分类栏、登录页和后台页面。

### 19.12 AI 设计输入

AI 设计阶段输入包括：

1. 图片预览图。
2. 图片主色调。
3. 图片内容识别结果。
4. EXIF 元数据。
5. 用户留言。
6. 用户是否启用 AI 文案。
7. 分类信息。
8. 可用模板列表。
9. `slide_templates.json` 中的模板能力说明。
10. `layer_primitives.json` 中的图层能力说明。
11. `ai_css_whitelist.json` 中的 CSS Tokens 白名单。

### 19.13 AI 设计输出

AI 输出包括：

1. 图片主题。
2. 图片意境。
3. 视觉主体。
4. 分类建议。
5. 主题色。
6. 模板选择。
7. 模板参数。
8. 图层配置。
9. 文案，视规则决定。
10. 时间轴样式。
11. 动画方案。
12. CSS Tokens。
13. 完整 `slide_designs.json`。

### 19.14 模板选择规则

AI 应根据照片内容选择模板。

示例：

- 山水、自然风光：优先 `poetic_landscape`、`cinematic_fullscreen`、`dark_exhibition`。
- 宠物近景：优先 `pet_portrait`、`warm_memory`、`soft_blur_background`。
- 家庭生活：优先 `warm_memory`、`quiet_album`、`soft_blur_background`。
- 构图高级的摄影照：优先 `gallery_center`、`dark_exhibition`、`minimal_white`。
- 色彩强烈的照片：优先 `cinematic_fullscreen`，但需要降低背景干扰。
- 低饱和纪实照片：优先 `quiet_album`、`film_negative`。

### 19.15 文案规则

文案字段包括：

1. 用户留言 `user_message`。
2. AI 文案 `ai_caption`。
3. 最终展示文案 `final_caption`。

规则：

```text
如果用户填写了留言：
    final_caption = user_message
    AI 不生成主展示文案，或生成但不展示。

如果用户没有填写留言，并勾选“AI 生成文案”：
    AI 生成 ai_caption
    final_caption = ai_caption

如果用户没有填写留言，也没有勾选“AI 生成文案”：
    final_caption = null
    幻灯片不展示主文案。
```

### 19.16 设计校验与质量评分

AI 生成设计后，系统必须执行两类检查。

#### 19.16.1 结构校验

1. JSON Schema 校验。
2. 模板 ID 校验。
3. 图层类型校验。
4. CSS Tokens 白名单校验。
5. 坐标范围校验。
6. 颜色格式校验。
7. 图层数量校验。

#### 19.16.2 审美质量检查

系统应尽量检查：

1. 文案是否遮挡主体。
2. 文字与背景对比度是否足够。
3. 蒙版是否过重。
4. 图层是否过多。
5. 背景是否抢主体。
6. 颜色是否过于杂乱。
7. 文案是否过长。
8. 页面是否存在明显视觉重心混乱。

如果检查不通过：

1. 优先让 AI 重新生成一次。
2. 仍失败则使用默认高质量模板。
3. 照片仍必须可播放。

### 19.17 AI 失败兜底

如果 AI 设计失败：

1. 使用默认模板 `gallery_center` 或分类默认模板。
2. 使用程序提取主题色。
3. 如果有用户留言，展示用户留言。
4. 如果没有用户留言，不展示文案。
5. 时间轴仍正常显示。
6. 照片仍进入可播放状态。

### 19.18 管理员重新生成规则

管理员可以对单张照片执行：

1. 重新生成全部设计。
2. 仅重新生成文案。
3. 仅重新选择模板。
4. 仅重新生成 CSS Tokens。
5. 仅重新生成图层配置。
6. 使用默认模板覆盖当前设计。

### 19.19 验收标准

1. 上传照片后能生成完整 `slide_designs.json`。
2. `slide_designs.json` 同时包含模板、模板参数、图层、CSS Tokens 和动画信息。
3. AI 不生成任意可执行 JavaScript。
4. AI 不生成任意 HTML。
5. CSS Tokens 必须通过 `ai_css_whitelist.json` 校验。
6. 前端能根据 `slide_designs.json` 渲染不同幻灯片风格。
7. 蒙版、形状、渐变、阴影、字体、时间轴均可被设计 JSON 控制。
8. JSON 与 CSS 不重复控制核心布局字段。
9. 用户留言优先于 AI 文案。
10. 未启用 AI 文案时不展示 AI 文案。
11. AI 失败时使用兜底模板。
12. 管理员可以重新生成设计。

## 20. AI Agent 工作流需求

### 20.1 Agent 定位

KinFrame 的 Agent 是“AI 幻灯片入库与设计 Agent”。

其职责不是自由聊天，而是完成照片入库过程中的分析、分类、文案和设计生成。

### 20.2 工作流步骤

```text
用户上传照片
  ↓
原图保存到 MinIO
  ↓
创建照片记录
  ↓
解析 EXIF
  ↓
生成预览图和缩略图
  ↓
提取主题色
  ↓
Ollama 视觉模型识别图片内容
  ↓
DeepSeek 生成分类建议、文案和设计方案
  ↓
生成 slide_design_json
  ↓
写入数据库
  ↓
照片进入 ready 状态
  ↓
前端 PPT 可播放
```

### 20.3 技术选型

AI 相关技术栈：

1. 本地 Ollama：视觉模型部署。
2. DeepSeek API：大语言模型，用于文案和设计方案生成。
3. LangGraph/deepagents：编排 Agent 工作流。
4. ExifTool：解析 EXIF 元数据。
5. Python 后端 Worker：执行异步任务。

### 20.4 状态机

照片处理状态建议：

```text
uploaded
processing
exif_parsed
preview_generated
vision_analyzed
design_generated
ready
failed
```

### 20.5 失败重试

1. EXIF 解析失败不应导致整个流程失败。
2. 预览图生成失败应进入 failed。
3. Ollama 识别失败时可使用默认分类或等待人工分类。
4. DeepSeek 失败时使用默认模板。
5. 任务应记录错误原因。
6. 管理员可以手动重试。

## 21. Slide Renderer 前端渲染器需求

### 21.1 定位

Slide Renderer 是 KinFrame 前端的核心组件。

职责：

- 接收当前照片数据。
- 接收 `slide_design_json`。
- 根据设计 JSON 渲染完整 PPT 页面。

### 21.2 组件结构

建议组件结构：

```text
SlideRenderer
├── SlideBackground
├── SlideImage
├── SlideCaption
├── SlideMeta
├── SlideTimeline
├── SlideDecorations
└── SlideTransition
```

### 21.3 渲染原则

1. 前端只接受通过 schema 校验的安全字段。
2. 不执行 AI 生成的 JavaScript。
3. 不直接插入 AI 生成的 HTML。
4. 页面结构由内置 Vue 模板和 Slide Renderer 控制。
5. 核心布局由 `template`、`templateParams` 和 `layers.position` 控制。
6. 视觉质感由 `styleTokens.cssVariables` 控制。
7. CSS Tokens 必须经过 `ai_css_whitelist.json` 校验。
8. 不支持未知模板时使用默认模板。
9. 不支持未知图层类型时忽略该图层并记录警告。
10. 不支持未知颜色格式时使用默认主题色。
11. 不允许 AI 控制顶部隐藏菜单栏、左侧分类栏、登录页、后台页面和路由逻辑。
12. 渲染器必须保证同一份 `slide_designs.json` 在不同浏览器中呈现基本一致的视觉效果。

### 21.4 动画要求

第一版支持：

1. 淡入淡出。
2. 轻微缩放。
3. 左右滑动。
4. 背景缓慢流动。
5. 文案延迟出现。

动画必须克制，不能影响照片观看。

### 21.5 验收标准

1. Slide Renderer 能加载不同模板。
2. 不同照片可以显示不同布局。
3. JSON 错误时能兜底。
4. 页面切换不卡顿。
5. 动画不破坏阅读和观看。

## 25. 非功能需求

### 25.1 性能

1. PPT 页面切换响应时间应小于 300ms，前提是图片已预加载。
2. 缩略图和预览图应优先加载，不直接加载原图。
3. 当前照片前后各至少预加载 1 张。
4. 外网访问时应尽量使用预览图。

### 25.2 安全

1. 必须登录后访问。
2. 不开放注册。
3. MinIO bucket 不公开。
4. 图片访问使用后端授权或短期 URL。
5. 密码必须哈希保存。
6. Session Cookie 应设置 HttpOnly。
7. 外网访问必须使用 HTTPS。
8. 上传文件需要校验 MIME 和扩展名。

### 25.3 兼容性

必须兼容：

1. Arch Linux 开发环境。
2. Windows + WSL2 开发环境。
3. Docker Compose 部署。
4. Chrome 浏览器。
5. Safari，尤其是 iPhone Safari。

### 25.4 可迁移性

1. 项目源码可通过 Git 迁移。
2. 数据通过 `data/` 目录或备份脚本迁移。
3. MinIO 数据可备份恢复。
4. PostgreSQL 可通过 dump 恢复。
5. `.env` 不进 Git，但需要手动备份。

## 29. 关键产品原则

1. 观看体验优先于管理体验。
2. 照片本身永远是视觉主体。
3. AI 设计必须被基础模板、模板参数、图层能力、CSS 白名单和 schema 共同约束。
4. 用户留言优先于 AI 文案。
5. 图片不能丢，原图必须保留。
6. 私密性优先于公开分享。
7. 全屏播放页是主入口，图片管理页是辅助入口。
8. 隐藏菜单必须好用，不能为了美观导致用户找不到功能。
9. 分类、模板、图层类型和 CSS Tokens 必须可扩展。
10. 项目必须支持未来从 Linux 迁移到 Windows + WSL2。

## 30. 结论

KinFrame 的核心产品方向是：

> 将家庭照片从“文件和图片列表”升级为“AI 设计的家庭影像艺术幻灯片”。

它的关键体验不是上传和管理，而是登录后立即进入一个沉浸式、全屏、优雅、可用方向键导航的家庭影像放映空间。

每一张照片都应像一页精心制作的 PPT 或摄影作品网页：背景、布局、文案、时间轴和动效都应服务于照片的情绪与故事。

本 PRD 的核心要求可以概括为：

```text
用户提供照片素材，AI 基于“基础模板体系 + 模板参数化 + 图层混合 + AI 生成式 CSS”生成幻灯片设计，前端稳定渲染，系统按分类和时间轴进行沉浸式放映。
```
